Open Source Tools Final Project 2012

Final Project

Due Sunday, December 16, 2012 11:59PM

Please stay tuned to this page for any clarifications

The goal of the final project is to demonstrate your knowledge of Open Source tools. In assignment 3, we wrote a command line tool to rank items, and in assignment 4, we put together a simple web-based interface to this system. For the final project, we will adapt this into a more full featured web-based application. First, some requirements for this project:

  1. It must be written in Python and deployed with App Engine.
  2. It must be developed using GIT, with regular code commits.
  3. It must store data in Google App Engine's data store.
  4. It must render HTML with templates.
  5. It must include documentation.

Given these requirements you will likely have to rewrite parts of your code (for instance, App Engine does not allow shell scripts to be invoked from Python).  In terms of new features, you should implement the following:

  1. The system should handle multiple users and associate categories/items and votes with a logged in user.
  2. Users should be able to create and edit a category with items via the web.  Anyone can vote on any category, but only the creator of a category can edit it.
  3. Users should be able to view any category via the web, not just the two fixed ones in assignment 4.
  4. Users should be able to export and import categories with XML.  Example format is here.  Exporting is possible with any user's categories.  Importing is possible only by the creator of a category.

In addition, you should implement one of the following advanced features.  It is up to you which one.

  1. Allow a user to change the items in a category.  When this happens, the results for that item must be removed.  Also, imports must be supported that replace a previous import (meaning items that do not change are preserved, items that are removed have results deleted).
  2. Support user comments about an item.  A user should not be able to comment on an item more than once.  The comments should be shown in the results view by each item.
  3. Search capability to find items and categories that match supplied keywords.  For example a search for "Apple" could match "Apple MacBook" in the category "Laptops" as well as "Apple" in the category "Fruits".
  4. Support for expiration time in categories, after which point voting is no longer possible but viewing results is.  Expiration times should be able to be set in the category editor.
  5. Support images associated with each item.  They should be displayed at vote time. In the editor, images must be uploaded into your data store (do not use links), and the XML format for both import and export must handle serializing images.

Note that you do not need to maintain data compatibility with the previous assignments, nor do you need to port any data over from the previous directory structure.  You will be storing your categories, items and votes with Google App Engine.  App Engine provides a built-in object store which you should use.

To learn more about App Engine, please be sure to read the following web pages:

  1. http://code.google.com/appengine/docs/whatisgoogleappengine.html
  2. http://code.google.com/appengine/docs/python/gettingstarted/
  3. http://code.google.com/appengine/docs/python/overview.html

FAQ:

  1. Can I write the project on my home computer?
    You may develop the project anywhere you want.  You can download the SDK for App Engine from the links listed above.  You can also download GIT here.  Be sure to hand in the .git repository and make sure you have tested your app before submitting.

  1. Can I use third-party software packages?
    You may use any third party software so long as it is freely available.  You must document in your developer guide what packages you use, if any.
     
  2. This project sounds stupid.
    That's not really a question. But you are welcome to pick your own project. If you want to do so, mail Prof. Korn to check if it's acceptable.
     

Documentation

All projects must include documentation. The documentation must cover two things:

  1. Any information the grader will need to be able to properly test your assignment.  If a feature is not obvious, it may be overlooked unless you explain it properly.
  2. A developers guide that explains how the project is designed and coded.  The grader will use this when evaluating the quality of your code.

Submission

Via the homework submission system, submit (1) your git repository, (2) your documentation, and (3) the project URL we can visit to try your project. The git repository should be created as a compressed tar archive of your .git directory, and should include no other files.  To create, you can do the following:

cd myproject
tar -cvf project.git .git
gzip project.git
# now submit project.git.gz

If you do not have tar, you can use zip:

cd myproject
zip -r project .git
# now submit project.zip

Do not use other formats/methods.
